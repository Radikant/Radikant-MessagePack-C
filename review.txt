Went through stream.c, encoder.c, decoder.c, zone.c, endian.h. Three real bottlenecks stand out, ranked by expected impact. All are drop-in, no API surface changes.


1. Decode always pays a zone-copy for str/bin/ext, even though a zero-copy path already exists and is just switched off

decoder.c:39-45 (and the bin/ext equivalents at 62-68, 91-97):


code:

{
if (decoder->zero_copy_strings && decoder->stream && decoder->stream->fast_left >= len) {
    out_obj->via.str.ptr = decoder->stream->fast_ptr;   // <- this path is dead by default
    ...
    return MP_OK;
}
out_obj->via.str.ptr = len > 0 ? (char*)mp_zone_alloc(decoder->zone, len) : NULL;
...
read_bytes(decoder, (void*)out_obj->via.str.ptr, len);   // <- alloc + memcpy every time		
}

mp_decoder_init (decoder.c:11) sets zero_copy_strings = false. That means every string/bin/ext payload gets a bump-alloc and a memcpy on top of the read, unconditionally — for a memory-buffer decode where the source buffer trivially outlives the parsed tree, that copy buys you nothing.

This is almost certainly why your own decode is so much slower than your own encode on the same data: SIMPLE VECTOR encodes at 771 MB/s but decodes at only 516 MB/s (33% slower doing strictly less logical work). mpack's node/tree API is zero-copy for str/bin/ext by default when reading from memory — its pointers refer directly into the source buffer. You're not comparing like-for-like; you're comparing "mpack's cheap path" against "radikant's defensive-copy path."

Fix: nothing to change in the library. Just flip decoder.zero_copy_strings = true before calling mp_parse_memory/mp_decode in the benchmark (and document that it requires the source buffer to outlive the object tree). This alone should close most of the SIMPLE/HARD decode gap. Optionally, expose it as a parameter on mp_parse_memory (decoder.h:29) instead of requiring manual decoder setup — small, non-breaking addition (default false to preserve current semantics).

2. Dynamic write buffer starts at 128 bytes and grows by doubling → realloc+memcpy tax on every encode call


code:
{
	size_t new_cap = ctx->capacity == 0 ? 128 : ctx->capacity * 2;
while (new_cap - ctx->size < count) new_cap *= 2;
char* new_data = (char*)realloc(ctx->data, new_cap);
}


Every call to mp_serialize_memory (encoder.c:254-271) starts a brand new ctx at capacity = 0. For anything bigger than ~128 bytes, you pay a realloc (malloc + memcpy of everything written so far) at every doubling step. For a payload that ends up at, say, 64 KB, that's 128→256→...→64K, ~9 reallocs, and the cumulative memcpy'd bytes across all those reallocs sum to roughly 2× the final size — pure overhead on top of the payload copy you actually need. That tracks with HARD VECTOR encode being ~17% behind mpack while DIFFICULT (probably smaller/flatter, fewer doublings) is actually ahead.

Fix, no API change: bump the initial capacity constant from 128 to something more realistic (1024 or 4096) so small/medium payloads hit it in one shot:

code:
{
size_t new_cap = ctx->capacity == 0 ? 4096 : ctx->capacity * 2;
}

If the benchmark harness controls the call site, an even better zero-cost fix is to call the existing non-dynamic-but-growable pattern — pre-size mp_stream_init_write with a capacity hint from a prior run/estimate, since the API already supports a preallocated buffer. That avoids touching the library at all.

3. Decode dispatch is a 5-way sequential range scan before it ever reaches the switch

decoder.c:120-172:

code:
{
if (b <= MP_TAG_FIXINT_MAX) {...}
else if (b >= MP_TAG_NEGFIXINT_MIN) {...}
else if (b >= MP_TAG_FIXSTR_MIN && b <= MP_TAG_FIXSTR_MAX) {...}
else if (b >= MP_TAG_FIXARRAY_MIN && b <= MP_TAG_FIXARRAY_MAX) {...}
else if (b >= MP_TAG_FIXMAP_MIN && b <= MP_TAG_FIXMAP_MAX) {...}
switch (b) { /* everything else */ }
}

Every single value byte pays up to 4 compare-and-branch pairs before falling through to the switch, which the compiler can turn into a jump table only for the tail. Since this runs once per decoded value (i.e. once per array/map element too), it's on the hottest path in the decoder. Flattening it into one dispatch, using GCC/Clang case-range extension (you already require __builtin_bswap*, so you're not targeting strict-ISO-only compilers anyway) lets the compiler build one jump table for all 256 leading bytes:

code:
{
	switch (b) {
    case 0x00 ... MP_TAG_FIXINT_MAX: /* positive fixint */ ...
    case MP_TAG_NEGFIXINT_MIN ... 0xff: /* negative fixint */ ...
    case MP_TAG_FIXSTR_MIN ... MP_TAG_FIXSTR_MAX: return read_string_to_zone(...);
    case MP_TAG_FIXARRAY_MIN ... MP_TAG_FIXARRAY_MAX: ...
    case MP_TAG_FIXMAP_MIN ... MP_TAG_FIXMAP_MAX: ...
    case MP_TAG_NIL: ...
    /* rest unchanged */
}

If you want to stay portable, a static uint8_t dispatch[256] lookup table computed once at compile time (or even a static const array literal) into a small enum of handler IDs achieves the same O(1) dispatch without compiler extensions. This is likely a smaller win than #1 and #2 but it's free and it's the one thing that scales with element count on both HARD and DIFFICULT vectors.

Minor, not worth chasing first

zone.c:22-48 abandons the unused tail of the current chunk whenever an allocation doesn't fit and starts a fresh fixed-size chunk — normal bump-allocator behavior, but chunk size never grows, so a decode with many mid-size allocations against a small chunk_size can rack up more malloc calls than necessary. If you find zone allocation shows up in a profile after #1–#3, doubling chunk_size on each new chunk (like the stream buffer does) is the same one-line fix.

Priority order to try: #1 first (it's a config flag flip in the benchmark, zero code risk, likely the single biggest number), then #2 (one constant), then #3 if you're still leaving MB/s on the table after profiling.
